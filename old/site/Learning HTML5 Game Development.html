Released

Discover how to create a game, modify and reuse game logic to extend your game, and optimize it for deployment. In this course, learn how to build two HTML5 games—both of which run on desktop and mobile browsers—using CSS3 and JavaScript ES6. As Makzan Hin shows how to build a counting game and a game in which players must run on floating platforms and maneuver through gaps and obstacles, he covers important concepts such as how to handle mouse clicking and tapping events, and make your game run smoothly on mobile displays. Plus, he explains how to create a gravity environment, adjust the difficulty of your game, and display scores.

Note: This course was created by Packt Publishing. We are pleased to host this training in our library.
Topics include:
  • Planning your first game
  • Setting up your file structure
  • Setting up HTML and JS
  • Drawing basic shapes with EaselJS
  • Implementing your first game object
  • Creating the core game logic
  • Adding graphics to your game
  • Planning your second game
  • Handling obstacle and hero collisions
  • Making the hero run and jump
  • Adding graphics and animations
  • Polishing your final game

- [Narrator] Hi there, welcome back to the video. In the last video, we implemented logic to check if the Hero is standing on any platform. In this video, we extend the GameObject to define moving logic. We will begin by extending the GameObject with moving logic. Then, we'll make the Hero to use the MovableGameObject. We will also focus the Hero at the center of the game screen whilst it is moving. Let's begin at the game+es6.js file in the project.

We create a new class named MovableGameObject. It extends GameObject class. In the constructor, we take two parameters. The first one is graphic, which follows the GameObject's constructor. The second one is a World reference, which we will need to access the other GameObjects in the World. That's because the moving GameObject will interact with other GameObject. We call this super with the graphic reference. Then, we use this.world to store the World reference.

We also define a velocity object, which contains both x and y velocity. For testing purpose, we give them value one. This is a movable object, so we need to have a loop to keep reflecting what's moving. In CreateJS, we can create game loop by handling the tick event. The tick event is fired from the stage to all its children in the entire display list. We have set up the original one in Game class. You can find the line of code that is, createjs dot Ticker dot on open brackets quotes tick quote comma this dot stage close bracket.

It essentially tells the CreateJS that we want to create a heartbeat for the stage container, which propagates the tick event to all the children. The reason we have set up the tick for this stage, is to re-render the stage 60 frames per second. According to the documentation, each time its tick method is called, it will render its display list to its target canvas. Inside the tick, we apply the x and y velocity to the GameObject. This code ensures that when we update the velocity value, the movement of the GameObject changes.

Let's move on to the Hero class. We now change the Hero to extend MovableGameObject instead of the original GameObject class. Let's move the World class. If we still have the testing code in the World class, we want to remove them at this moment. Now, we want to modify the World position to focus on the running Hero. When the Hero runs from left to right, we don't want the Hero's graphic to go outside the game screen.

That's because the focus is that Hero. We can move the World based on the Hero's x velocity. It is a technique known as Camera. In our game, we only need to follow the Hero in x axis. But, if the game allows the Hero to move in y axis, and beyond the game screen, the Camera also need to handle the y axis too. When the subject moves in one direction, we move the entire container in the opposite direction. Let's see what happens if we don't follow the opposite direction of the Hero.

We can disable the falling of the Hero by setting the y velocity to zero. Then, we disable the World's moving code in the tick method. Save the file, and refresh the browser. We can see the Hero is moving towards the right and disappears at the game's boundary. Now, if we move the World, but in a different speed of the Hero's velocity, let's say we move the World in a negative two value. Refresh the browser, and now the whole World disappears because the Camera is moving too fast.

Now, if we set the World to follow the same velocity of the Hero in positive value. Refresh the browser, and we can see both the Hero and the World is moving forward, and the World disappears after a short while. Let's revert back to our code and make the World move in a negative direction with the same speed of the Hero's x velocity. Refresh the browser, and now the World moves backward while the avatar moves forward. It is like we have a Camera focusing on the Hero and tracing its movement.

We have defined the Hero as a MovableGameObject. We also focused on the Hero's movement at the center of the screen when it moves. In the next video, we will make the Hero falls by adding gravity to the World.
